# Git 工作流

本文档定义本仓库的 Git 协作细则。只有在执行 commit、branch、push、merge、rebase、stash、冲突处理或 PR 准备时才需要读取。

## 目标

- 让代理承担日常 Git 检查、暂存、提交组织和风险提醒。
- 让用户专注于需求、设计和代码意图。
- 所有会影响远端、历史或用户未提交改动的操作，必须由用户确认后执行。

## 代理默认负责

- 在编辑、提交、切换分支或创建分支前，检查 `git status --short`。
- 自动区分当前任务相关改动和无关改动。
- 只暂存当前任务相关文件；不要使用 `git add .`。
- 提交前检查 `git diff --cached`。
- 按任务意图生成清晰的中文 commit message。
- 发现工作区有未提交改动时，先判断是否与当前任务相关。
- 发现潜在风险时主动提醒，例如未提交改动、冲突、误带文件、需要 rebase、需要 force-push。

## 主动提醒

- 如果用户开始一项新任务但当前不在合适分支，提醒是否需要创建或切换分支。
- 如果当前任务已完成且存在相关未提交改动，提醒是否需要 commit。
- 如果本地提交尚未推送，提醒是否需要 push。
- 如果工作区存在未提交改动且用户准备切换任务，提醒是否需要 commit、stash 或创建新分支。
- 如果当前分支与默认集成分支可能落后，提醒是否需要同步，但不要自动 pull、merge 或 rebase。
- 提醒时给出简短建议和可选操作，不要打断正在进行的代码修改流程。

## 用户确认边界

代理应主动提醒以下操作，但执行前必须得到用户确认：

- 创建、切换、重命名或删除分支。
- commit。
- push。
- pull、merge、rebase、squash。
- stash 或恢复 stash。
- 删除、丢弃、回滚、覆盖任何本地改动。
- force-push，即使使用 `--force-with-lease`。
- 解决包含业务取舍的 merge conflict。

## 默认禁止

- 不要执行 `git reset --hard`。
- 不要执行 `git clean -fd`。
- 不要使用破坏性 checkout 覆盖文件。
- 不要把无关改动带入当前提交。
- 不要重写历史或修改远端状态，除非用户明确要求。

## 分支策略

- 一个任务或一个 PR 对应一个分支。
- 分支命名使用小写 kebab-case：
  - `feat/<scope>-<summary>`
  - `fix/<scope>-<summary>`
  - `docs/<scope>-<summary>`
  - `refactor/<scope>-<summary>`
  - `chore/<scope>-<summary>`
- 创建或切换分支前，确认当前分支和工作区状态。
- 工作区存在未提交改动时，不要擅自切换分支。
- 未经用户明确批准，不要 rebase、squash、删除分支或 force-push。
- 如果用户明确批准 force-push，优先使用 `git push --force-with-lease`。

## 提交组织

- 提交前先分析当前改动，按意图、模块和风险拆分为有意义的提交。
- 不要把所有改动一次性提交，除非这些改动共同表达一个不可拆分的单一意图。
- 一个提交只表达一个清晰意图。
- 优先拆分为：行为变更、测试、文档、配置、机械格式化、重构等独立提交。
- 不要把无关重构、格式化、文档和行为变更混在同一个提交里。
- 提交前检查暂存区，确认没有误带无关文件。
- commit message 使用中文，简洁描述本次变更意图。
- 如果团队使用 Conventional Commits，可使用中文摘要：
  - `feat: 添加用户登录入口`
  - `fix: 修复接口超时重试`
  - `docs: 更新部署说明`
  - `test: 补充订单状态测试`
  - `refactor: 简化配置加载流程`
  - `chore: 调整构建脚本`

## Stash 策略

- 不要自动 stash 用户改动。
- 只有当用户确认需要临时切换上下文时，才使用 stash。
- stash 前说明将保存哪些改动。
- 恢复 stash 后检查是否产生冲突或混入无关改动。

## 同步策略

- 不要自动 pull、merge 或 rebase。
- 如果当前分支可能落后，先提醒用户并说明可选方案。
- 如果仓库已有明确策略，优先遵守仓库策略。
- 如果没有明确策略，在 merge 和 rebase 之间选择前先询问用户。

## 冲突处理

- 不要机械选择 ours/theirs。
- 先说明冲突文件、冲突区域和可能的业务含义。
- 能根据代码意图安全解决的冲突，可以提出方案并等待用户确认。
- 涉及业务取舍、数据迁移、接口兼容或安全边界时，必须由用户确认。
- 解决冲突后运行相关验证命令。

## Push 和 PR 准备

- 不要自动 push。
- 用户确认 push 后，新分支优先使用 `git push -u origin <branch-name>`。
- push 前确认当前分支、暂存区、提交历史和未提交改动状态。
- 准备 PR 时，提供变更摘要、验证命令和已知风险。
- 如果本地有提交未推送，应提醒用户是否需要 push 或准备 PR。
